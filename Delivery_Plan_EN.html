1Project Overview

The project designs, implements, integrates, tests, deploys and supports a GenAI platform for Alinma Bank that combines three coordinated capabilities:

  • M1 — AI Outbound Telesales
  • M2 — AI Follow-up for Digital Leads
  • M3 — AI-First Inbound Contact Center (Phase 1 inquiry, Phase 2 complaints)

Total delivery duration is 5 months. All AI workloads and customer interactions run on bank-controlled on-premises infrastructure within KSA.

2Scope of Delivery

2.1 Our Core Deliverables (within 5 months)

  • Full platform software stack: AgentStudio (low-code agent builder + LLM inference engine), Voice SaaS, ASR / TTS (Arabic + English), RAG pipeline + knowledge base management, change-governance capability, integration consumer-side code, data contracts + EDW push.
  • Three UAT Bots (one each for M1 / M2 / M3 Phase 1): end-to-end, production-ready, serving as the skeleton version of the full bots (see 2.2).
  • Training for 4 audiences (see §9.1).
  • Hypercare: post-go-live extended support.
Change-governance capability: changes to bot-configurable items (dialog flow, prompts, knowledge content, model selection, etc.) are managed through diff review + maker-checker dual approval + version control + rollback + audit logging. The detailed governance scheme (approval levels, coverage) is confirmed with the Bank during implementation.

2.2 UAT Bot Approach

We deliver the platform tools plus 3 UAT Bots (one each for M1 / M2 / M3 Phase 1). Each bot completes an end-to-end flow over a minimal integration footprint (ESB + key business services + IVR/SIP + ASR/TTS/LLM + knowledge base + human handover), serving both as end-to-end acceptance evidence and as the initial version of the production bot. Each UAT Bot is designed jointly with the business unit in P2, covering three dimensions: business scenario, integration targets (T24/CRM/Comarch), and Identity & Verification (ID&V).

ID&V (Identity & Verification) is a critical step in the voice channel. Per RFP INT-27, authentication logic is provided by the Bank’s integration-layer services; the AI agent does not implement its own authentication. During implementation we will conduct targeted discovery and design of the inbound / outbound authentication logic, based on the Bank’s existing ID&V capabilities and intentions.

8Key Assumptions

The plan is premised on the following. If any does not hold, a change-control process is triggered for re-assessment.

Scope boundary

  • Full bots for M1 / M2 (all product lines) and M3 Phase 2 are built by Alinma on our platform (see 2.3).
  • The client already has its own outbound dialing + omni-channel marketing system (presumed to include Comarch); we integrate, not replace.
  • Business rules (eligibility / transfer / complaint taxonomy / mandatory-human) and knowledge content are provided by the Alinma business unit.
  • Alinma’s existing IVR / contact center / dialer can be integrated (no replacement / re-build; SIP-level integration only).
  • Multi-step business orchestration is handled by Alinma’s existing orchestration capability or integration layer; AgentStudio handles dialog-flow orchestration only.
  • EDW-side ETL / modeling / reporting / archival are handled by the Alinma EDW team; we provide data contracts and push.
  • Hardware bill of materials and procurement are provided by the local SI.

Client readiness

  • Alinma already has an ESB + coarse-grained business services + shared logging / security / error-handling services.
  • Alinma already has IAM/SSO (OAuth2/OIDC), KMS, SIEM, APM and other shared infrastructure.
  • Alinma provides real call samples (Saudi dialect, annotated) for language-capability evaluation and demonstration (not for model training).
  • Alinma has approved internal budget and headcount for the project.

Time & environment

  • Actual project start date ≤ 01 Sep 2026 (to avoid Ramadan 2027).
  • Development environment uses remote access from the overseas team to the in-KSA development environment (to be aligned with the Bank’s compliance).
  • ASR / TTS / LLM are used as-is at production-grade capability; this project includes no model training or tuning on client data. Language capability is evaluated on actual performance, with acceptance criteria agreed jointly at test time against global benchmarks (per RFP §12).

A.3 Reasonableness cross-checks

  1. Waterfall staffing curve: P1 ~2.75 people (PM + architect + light discovery), P2 ~5, P3 peak ~7.7 (build phase), P4–P5 ~8–9 (test/UAT), P6 ~6.5 — staff phase in and out, no “full team throughout” padding.
  2. Integration 190 person-days (23%, single largest) — reflects deep banking integration (ESB + CRM + T24 + Comarch + IVR + auth, 6 systems, under INT constraints); no engagement in P1, only the integration lead joins contracts in P2 (0.5 FTE), full load in P3–P4 (3.0 FTE), defect-fix in P5–P6.
  3. P3 staffing peak 230 person-days (28%) — consistent with “infrastructure + platform deployment + integration” being the critical path.
  4. Team size: P3 peak ~7–8 concurrent, delivery-period average ~6.5 (710 ÷ 110 working-days) — a lean, reasonable size for a 5-month, 3-module, high-compliance banking project.
  5. Testing & compliance back-loaded: QA largely absent in P1–P2, rising to 1.3–1.7 FTE in P4–P5 (SIT/perf/UAT; no repeated WER testing since no model training); compliance rises to 1.67 FTE in P5 (review materials + hardening).
  6. Dialog / Bot Build 120 person-days (15%) — combines dialog design + KB + RAG + change governance (WF3) and the build of 3 UAT Bots (coverage maps to RFP §12 UAT, see 2.2).

This is a v1 top-down estimate (±15%). After award it is re-estimated bottom-up against a detailed WBS, incorporating team productivity factors and actual client readiness (e.g. whether coarse-grained services must be newly built, which materially affects integration effort).
